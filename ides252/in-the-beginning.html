The Internet!

I’d like to start by clarifying the distinction between the Internet and the World Wide Web. The Internet was invented in the 1960s by an American military research group named the Defense Advanced Research Projects Agency (DARPA)—yes, we have the US military to thank for the Internet. This was during the Cold War, when the US was focused on defending themselves from a nuclear missile attack by the Soviet Union. They were working on a number of critical projects at military and university research facilities located across the US, collaborating by sharing information through what were essentially phone lines—if a lab in Vermont, let’s say, had to share a file with a lab in New Mexico, they would create a direct wire connection through switches located along a route between the two labs (all phone calls worked that way at the time: physical wires were connected by mechanical switches).

The heightened anxiety stemming from nuclear threat led scientists to realize that this communications system was vulnerable: a nuclear strike at the right location could cut off communication between two collaborating labs completely, hobbling the US’ progress in the arms race. So a new way of sharing computer information was invented, in which the file or data to be shared was divided into small chunks, and each chunk was given the address of the intended destination. These “packets” were then sent into the network, where they would travel from switch to switch until they arrived at their destination, to be reassembled into the original file—and the route they took didn’t matter. This meant that single nodes of the network could be destroyed without harming the data’s capability to move through any part of the network to reach its destination. This new kind of data transmission was enabled by TCP/IP (Transmission Control Protocol/Internet Protocol): its invention was essentially the invention of the Internet, and it powers your computer’s Internet connectivity today.

So the Internet is a system of physical connections—wires, fibre optic cables, microwave antennas, satellites, routers, switches—and protocols like TCP/IP that allow computers to communicate over those connections.

The World Wide Web

The World Wide Web (let’s just call it the web from now on) was born much later, in 1990. Interestingly, it was invented by one guy: one extraordinary and legendary guy. Tim Berners-Lee was an English physicist working at a particle physics laboratory in Switzerland when he (and his team—OK, maybe it was more than one guy) invented HTTP (Hypertext Transfer Protocol) and HTML (Hypertext Markup Language), the building blocks of the web . The web of 1990 was just the latest of many services that ran on the Internet: there were already several Internet services available such as email, file transfers and newsgroups. The web was really a very simple service: you used a client program called a “browser” (think Chrome) to request a file from a server program, which then sent you the file. The big breakthrough built into HTML was that parts of that file could link to other files, allowing you to browse with ease (or “surf,” although no-one has adequately explained how you surf through a web) from file to file in a linked web of information.

Now, of course, the web is the most important communication medium in the world. Well done, Mr. Berners-Lee.

We’re not going to learn anything about HTTP in this course; that stuff is for back-end web developers (more about that below). We will be learning a thing or two about HTML though.

The URL

When we refer to bits of HTML “linking” to other files on the web, we’re talking about our old friend the Uniform Resource Locator. This helpful little item provides a way to describe the exact location of every single file on the Internet—every email message, web page, video, designer’s CV in PDF form—every file, of the trillions of files out there, has a unique URL. You encounter these every day, most often in the address bar at the top of your browser window. Let’s examine a URL:

http://trailers.apple.com/trailers/independent/dontbeafraidofthedark/index.html

URLs generally consist of these four parts:

  • the protocol (the http:// part in this case, since it’s an HTTP URL; in other cases, it may be feed:// or ftp:// or file:// or another protocol)
  • the host address (the trailers.apple.com part)
  • the directory structure (the nested folders /trailers/independent/dontbeafraidofthedark/— these are optional, since the file might be right at the top [“root”] level of the server rather than in a folder)
  • the filename (index.html)

The host address has its own technical wrinkle. Because TCP/IP finds a computer or device on the Internet through its numerical IP address (you’ll find your computer’s IP address in the Network part of your System Preferences), when you point your browser to trailers.apple.com, it needs to connect to 77.67.85.10 to get the content. That’s great for computers—they love numbers!—but an address like that is not very people-friendly. Fortunately, your browser can make use of the Domain Name Service (DNS), which takes the host address you typed in and looks it up in a giant database to find the required IP address. That’s one of the first things your browser must do before you can access any website, and if DNS is ever unavailable for any reason, your browser thinks the whole web has disappeared.

URLs have some restrictions as far as the characters that can be used in them (more on why that’s important later):

Disallowed characters in URLs: TAB SPACE " < > [ ] \ ^ ` { } |
Special characters in URLs: % # / ? : ; @ = & ~ +

The Site!

The websites you create will simply consist of a folder containing some files (and likely other subfolders). The folder structure of your site can be an aid to organization and production. Folders should have simple one-word names (or multi-word names can be faked by replacing spaces with the underscore character—see “disallowed characters” above). In the root directory of your site (“root” is the common name for the top-level directory of a website) you’ll place an HTML document named “index.html”—this is the homepage, the first page visitors see when they arrive at your site. NOTE: This filename is set in stone, and is used because Unix-based HTTP servers are conventionally configured to look for a document called “index.html” when no filename is specified by the browser making the request. When you ask Safari to go to fertilizer.ca for instance, the server returns the document at the URL http://fertilizer.ca/index.html.

You’ll probably also have several other HTML documents in that folder, although you might choose to place them inside folders which you’ve given names appropriate to the section of your site to which those documents belong. This is for your own convenience, since a large site can become a mass of files in no time and it gets difficult to find what you’re looking for. NOTE: Every file must have a filename extension, such as .html, .css, .js, .gif, .png. Folders shouldn’t have a filename extension.

You’ll also likely have a folder named “images.” Inside this folder, you’ll place all of the image files referenced by index.html and by any other pages. You might also place folders for images inside the organization subfolders discussed above, and have different sections’ images in different places. By the way: using short names for files and folders leads to a small savings in download time for your site. Instead of “images” for the folder name, why not use “i” or “img”? It’s not a problem to name the folder that way; “images” is not set in stone.

The Back End!

I tried to think of a one-word descriptor for this section, but I failed—but this stuff is universally known as the “back end” anyway. The back end refers to technological solutions to the needs of very large and/or very frequently updated websites. A collection of HTML files and image files is fine for a very small website, but think of creating a site like amazon.com—there are hundreds or thousands of items there, with new ones being added (or removed) daily. They also have information about thousands of customers, allowing them to log in, add shipping addresses, add items to their shopping cart, etc. When a website needs this kind of complex content and functionality, you’d need a gigantic army of well-trained, poorly paid web page authors to manage it if it were made up of static HTML files.

Instead, the content of the site—every page—is abstracted out of the HTML markup and stored in a powerful database running on the server. When a visitor makes a request for a page—homepage, product category page, product detail page, search results page, address book page, any page—a program running on the server requests the needed content from the database, plugs it into an HTML template, and then sends the results off to the visitor’s browser as an HTML document (the browser has no idea that the page didn’t start out as a static file).

As I mentioned, a database is required to accomplish this kind of power: the most common database server on the web is MySQL (http://mysql.com/), largely because it’s open source (free!) and excellent. Accompanying the database, a programming language is used to create the functionality that works with the database to add, modify and retrieve content: the most commonly-used language is PHP (http://php.net/—it stands for PHP: Hypertext Pre-Processor; don’t ask) since it’s open source and excellent.

Designing for the kind of websites we’ve just discussed requires anticipating and accommodating frequently-changing content, building the necessary flexibility into the design: chunks of the page may need to get taller or shorter, or more or fewer of certain elements may need to be displayed, as the visitor views different parts of the site. Generally speaking, designers need to embrace change and flexibility when designing for the web, for many reasons, particularly this one: the very vehicle for the design—the browser window—is a flexible environment beyond our control. Users can choose to change their browser window width and height, their type sizes and faces, and colours. Designing websites is often more like designing software than designing a poster or magazine. And believe it or not, if you decide you want to become a web designer, you will come to see that as a good thing.

The Front End!

As you know from your work in your UX course, the web design is traditionally begun as a sketch, then worked into wireframes using a tool like OmniGraffle or Illustrator, then turned into a set of polished visual mockups using Photoshop or Fireworks. Once those are completed and approved by the client, production on the site begins: images are exported as the smallest, most highly optimized versions possible (more on that later in the course), and the necessary pages are created using HTML, CSS (Cascading Style Sheets) and JavaScript, the “three-legged stool” of web design:

(thank goodness it’s not a two-legged stool: those things are really tippy)

HTML is, as previously mentioned, a markup language that provides the structural skeleton of the content: its purpose is to define the role or meaning of each part of the document, through the use of tags. For instance, a paragraph tag, with content between an opening and closing p tag:

<p>This is a paragraph</p>

It declares such things as: a certain chunk forms the page’s masthead; another bit forms the footer; this chunk provides the page’s primary navigation, consisting of a list of links; this bit is the main heading for the page; this is a sub-heading for this section; this is a login form; this is a caption for an image—hopefully you get the idea. A big part of learning HTML is learning the correct tag to use in a particular situation, known as the semantics of HTML. HTML is also used to create links, and to bring images and other media into the document’s content.

What HTML is not meant to do is to define how things look: that’s the job of CSS. Cascading Style Sheets has been around for a long time (CSS2 was released in 1998); it’s a presentation language that was invented as a way of giving web designers some real power above and beyond the ugly HTML hacks that they relied on during the 90s. CSS provides control over layout, type styling and formatting, and background images, through the use of rules:

p	{
	margin: 0;
	padding: 0;
	color: #7F0026;
	background-color: #DDDDDD;
}

This is a CSS rule. The part before the opening curly brace is known as the selector — this part defines what part of the page gets the style (in this case p, i.e., the HTML paragraph element). The part within the curly braces is a declaration block. This consists of, in this case, four declarations. Each declaration consists of a CSS property and its value, separated by a colon and ending with a semicolon. This rule states that a paragraph’s background should be light grey, the text colour should be dark red, and the paragraph itself should have no padding or margins around its four sides. Don’t let those funky codes for the colours worry you: we’ll be using Dreamweaver’s and Photoshop’s help to get those. We’ll be authoring pages using CSS2 for the most part, with maybe a sprinkle or two of CSS3.

JavaScript is a programming language that is used to add or enhance behaviour of elements in a web page. When you type “tarsier” into Google’s search field and a vertical list of clickable options appears under the field, it’s JavaScript doing that. JavaScript is a bit outside of the scope of this course, although we’ll probably look at a few easy JavaScript tricks that Dreamweaver can do.
